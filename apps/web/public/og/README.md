# Open Graph images

OG images are static **1200×630 PNG** files. SVG is **not** supported by most social crawlers (Facebook, iMessage, WhatsApp, LinkedIn) — always export to PNG before referencing in `<meta property="og:image">`.

## Files in this folder

| File | Purpose |
|---|---|
| `home.svg` / `home.png` | Default OG image for the landing page. |
| `og-fallback.png` (in `/public`) | Final fallback if a page-specific OG is missing. |
| `{festival-slug}.png` | Festival-specific OG (16 originals: mad-cool, primavera-sound, sonar, fib, bbk-live, resurrection-fest, arenal-sound, medusa-festival, vina-rock, o-son-do-camino, cala-mijas, sonorama-ribera, zevra-festival, low-festival, tomavistas, cruilla). Referenced by `sitemap-festivales.xml` via the `FESTIVAL_OG_IMAGES` set. |
| `templates/*.svg` | **Per-page-type SVG templates with `{{PLACEHOLDER}}` tokens.** See below. |

## SVG templates (per page type)

The `templates/` folder contains five reusable SVG templates — one per content type. Each is a flat 1200×630 SVG with `{{TOKEN}}` placeholders that get replaced with real content before exporting to PNG.

| Template | Used by | Placeholders |
|---|---|---|
| `festival-template.svg`   | `/festivales/{slug}`, `/como-llegar/{slug}` | `{{FESTIVAL_NAME}}`, `{{FESTIVAL_DATES}}`, `{{FESTIVAL_CITY}}`, `{{PRICE_FROM}}` |
| `route-template.svg`      | `/rutas/{origen-festival}`                  | `{{ORIGIN}}`, `{{DEST}}`, `{{DISTANCE}}`, `{{TIME}}`, `{{PRICE}}` |
| `dataset-template.svg`    | `/datos`, dataset CSV/JSON detail pages     | `{{DATASET_TITLE}}`, `{{DATASET_YEAR}}`, `{{ROW_COUNT}}` |
| `pillar-template.svg`     | `/guia-transporte-festivales`, `/como-funciona`, pillar pages | `{{GUIDE_TITLE}}`, `{{GUIDE_SUBTITLE}}`, `{{GUIDE_TAG}}` |
| `blog-template.svg`       | `/blog/{slug}`                              | `{{POST_TITLE}}`, `{{POST_CATEGORY}}`, `{{POST_AUTHOR}}`, `{{POST_DATE}}` |

Each template has a companion `*-demo.png` (1200×630) in the same folder showing what the rendered output looks like with sample values. Demos are regenerated by running:

```bash
node apps/web/scripts/gen-og-template-demos.mjs
```

All templates share the ConcertRide brand:
- Background `#080808` with light noise filter
- Primary lime `#D4F700`, secondary orange `#FF4F00` (gradient accents)
- Display: Archivo Black, Body: Inter, Mono: JetBrains Mono
- Corner accents in primary lime
- Footer with `concertride.me` URL

## How to generate a PNG from a template

There is **no runtime/dynamic OG generation** today — we do not run `@vercel/og` or `satori` in the Worker (bundle-size constraint on Cloudflare Workers, plus the SPA renders most pages client-side). PNGs are generated **once, manually, at build time** when a new festival/dataset/pillar page is added.

### Option A — Inkscape (CLI, recommended for batch)

```bash
# 1. Copy the template and replace tokens with sed (or any text editor)
cp apps/web/public/og/templates/festival-template.svg /tmp/mad-cool.svg
sed -i 's/{{FESTIVAL_NAME}}/MAD COOL/g; s/{{FESTIVAL_DATES}}/10–12 JUL 2026/g; s/{{FESTIVAL_CITY}}/MADRID · IBERDROLA MUSIC/g; s/{{PRICE_FROM}}/5/g' /tmp/mad-cool.svg

# 2. Export to 1200x630 PNG
inkscape /tmp/mad-cool.svg --export-type=png --export-filename=apps/web/public/og/mad-cool.png --export-width=1200 --export-height=630

# 3. Optimize (optional, ~30% smaller)
pngquant --quality=70-90 --force --output apps/web/public/og/mad-cool.png apps/web/public/og/mad-cool.png
```

### Option B — sharp (Node, runs in CI)

```bash
npx sharp -i /tmp/mad-cool.svg -o apps/web/public/og/mad-cool.png resize 1200 630
```

### Option C — Figma / Browser screenshot (one-off, manual)

Open the SVG in a browser, take a 1200×630 screenshot, save as PNG.

## When to generate a new PNG

Add a new PNG to this folder when you:
- Add a new festival landing page (and want better social sharing than the home fallback)
- Publish a new dataset on `/datos` and want a unique social card
- Launch a new pillar guide (`/guia-…`)
- Publish a flagship blog post with significant social distribution

For programmatic routes (44 cities × 16 festivals = 704 route pages) and 67+ artist pages, **do NOT** generate per-page PNGs — the fallback to `home.png` is acceptable. The cost (storage + maintenance) outweighs the benefit.

## Naming convention

`{type}-{slug}.png` or just `{slug}.png` if the type is unambiguous:
- Festivals: `mad-cool.png`, `primavera-sound.png`
- Datasets: `dataset-{slug}.png` (e.g. `dataset-precio-medio-carpooling-vs-bus.png`)
- Pillars: `pillar-{slug}.png` (e.g. `pillar-guia-transporte.png`)
- Blogs: `blog-{slug}.png` (rare — only for featured posts)

## Wiring up in code

After exporting a PNG, reference it in code:
- Festival: add the slug to `FESTIVAL_OG_IMAGES` Set in `apps/web/scripts/seo/generate-sitemaps.mjs` so the sitemap includes `<image:image>` tags.
- Page-specific override: pass `ogImage: "/og/{slug}.png"` to `useSeoMeta()` on that page.
- Schema: set `image: "${SITE_URL}/og/{slug}.png"` in any JSON-LD `Article`/`Event`/`Dataset` schema on the page.

If you do not provide a page-specific image, the default `<meta property="og:image" content="${SITE_URL}/og/home.png">` from `index.html` is used.
