# Service Worker — Caching Strategy (2026-05-20)

Source: `apps/web/src/sw.ts`. Built by `vite-plugin-pwa` in `injectManifest` mode
(see `apps/web/vite.config.ts`). Output: `dist/sw.js`.

The SW combines:

1. A small **precache** (app shell + critical icons) generated by Workbox from
   `globPatterns` at build time.
2. **Runtime caches** registered per URL pattern, with different strategies and
   TTLs tuned to how often each page type changes.
3. An **offline fallback** (`/offline.html`) served for any failed navigation
   request.

---

## Strategy by URL pattern

| Pattern                                              | Strategy           | TTL      | Max entries | Cache name           | Why                                                                 |
| ---------------------------------------------------- | ------------------ | -------- | ----------- | -------------------- | ------------------------------------------------------------------- |
| `/festivales/*`, `/festivales-en/*`, `/rutas/*`, `/conciertos/*` | CacheFirst         | 7 days   | 200         | `cr-landings-v1`     | Prerendered landing pages, content stable for days at a time        |
| `/datos/*` (HTML + JSON/CSV)                         | CacheFirst         | 30 days  | 30          | `cr-datasets-v1`     | Static datasets, change at most monthly                             |
| `/blog/*`                                            | NetworkFirst (3 s) | 1 hour   | 150         | `cr-blog-v1`         | Blog posts may get hotfixes; fall back to cache if offline/slow     |
| `/api/concerts`, `/api/festivales`, `/api/festivals`, `/api/venues`, `/api/rides` (GET only) | StaleWhileRevalidate | 5 min    | 100         | `cr-api-concerts-v1` | Cheap perceived load; background refresh keeps data current        |
| `/api/auth/*`, `/api/push/*`, `/api/messages/*`, `/api/users/me`, `/api/admin/*`, `/api/reports/*` | network only       | -        | -           | -                    | Auth + user-specific + mutation endpoints must never be cached      |
| `/assets/*`, `/images/*`, branded icons              | CacheFirst         | 90 days  | 300         | `cr-static-v1`       | Vite hash-named bundles, safe to cache long; icons rarely change    |
| `/og/*`                                              | CacheFirst         | 30 days  | 100         | `cr-og-v1`           | OG share images are stable per route                                |
| `https://fonts.googleapis.com/*`                     | StaleWhileRevalidate | 30 days  | 20          | `cr-fonts-v1`        | CSS index — updates eventually pick up new font hashes              |
| `https://fonts.gstatic.com/*`                        | CacheFirst         | 1 year   | 50          | `cr-fonts-v1`        | Actual woff2 binaries, content-addressed                            |
| `/offline.html`                                      | precache (own cache) | -        | -           | `cr-offline-v1`      | Always available even if everything else is cold                    |

Write methods (POST/PUT/DELETE/PATCH) are never matched — the route conditions
explicitly require `request.method === "GET"`. The same is true for any URL
under `/api/auth/*` regardless of method.

Ticketmaster image hosts (`s1.ticketm.net`, `media.ticketmaster.com`, etc.) are
intentionally **not** matched by any route — they're served straight from the
TM CDN and the browser HTTP cache handles them. This aligns with the TM ToS:
ConcertRide never hosts or re-distributes TM images.

## Cache namespaces

All runtime caches use the `cr-*-v1` prefix and are declared in
`CACHE_NAMES` inside `sw.ts`:

- `cr-landings-v1`
- `cr-datasets-v1`
- `cr-blog-v1`
- `cr-api-concerts-v1`
- `cr-static-v1`
- `cr-og-v1`
- `cr-fonts-v1`
- `cr-offline-v1`

In addition, Workbox manages its own precache cache (named automatically based
on the build hash) — it's busted whenever the precached file list changes.

## Cache invalidation after deploy

The SW uses three mechanisms together:

1. **Precache hash** (automatic). Every time a file in `globPatterns` changes,
   Workbox writes a new manifest. `cleanupOutdatedCaches()` then deletes the
   old precache during `activate`.
2. **`SW_VERSION` constant** in `sw.ts`. Bump this when you change cache shapes
   (rename a namespace, change strategy semantics, etc.). The `activate`
   handler deletes any `cr-*` cache that's not in `KNOWN_CACHES`, so renaming
   `cr-landings-v1` → `cr-landings-v2` and adding the new name to
   `CACHE_NAMES` is enough to evict the old cache cluster-wide.
3. **`registerType: "autoUpdate"`** in `vite.config.ts` keeps Workbox checking
   for a new SW on every page load. Combined with `skipWaiting()` +
   `clients.claim()` in the `install`/`activate` handlers, an updated SW takes
   over within one page navigation.

The page can also force an update by posting `{ type: "SKIP_WAITING" }` to the
waiting SW (handler is in `sw.ts`).

## Offline behavior

When the network fails for a **navigation** request (top-level page load), the
SW serves `/offline.html` from the `cr-offline-v1` cache. This file is:

- Precached during `install` via `caches.open(cr-offline-v1).add(...)`, so it's
  available even on the very first install before the Workbox manifest runs.
- Also included in the Workbox `globPatterns`, so it survives precache rebuilds.
- Self-contained: no external CSS, no Google Fonts dependency, no JS, no
  Ticketmaster references. Uses system font stack and inline styles so it
  renders identically online and offline.

Failed sub-resource fetches (images, JS chunks, font files) do **not** trigger
the offline fallback — they propagate as normal network errors. The cached
shell + precached chunks generally let the app boot offline anyway.

## Testing the SW

### Dev (vite + workbox-window)

The PWA plugin registers the SW in production builds. To test in dev:

```bash
cd apps/web
npm run build
npm run preview
```

Open `http://localhost:4173` in Chrome. The SW only registers over `https` or
`localhost`.

### Chrome DevTools

1. Open DevTools → **Application** tab.
2. **Service Workers** panel:
   - Verify the SW is `activated and running`.
   - Tick **Update on reload** during dev — re-fetches the SW on every reload.
   - Use **skipWaiting** when a new SW is `waiting`.
3. **Cache Storage** panel:
   - You should see `cr-landings-v1`, `cr-datasets-v1`, `cr-blog-v1`, etc.
   - Click an entry to inspect cached responses (URL, headers, body).
4. **Manifest** panel: confirms `manifest.webmanifest` is valid and the app is
   installable.

### Offline test

1. DevTools → **Network** tab → set throttling to **Offline**.
2. Navigate to a page you haven't loaded yet — should serve `/offline.html`.
3. Navigate to `/festivales/mad-cool` after loading it once online → it should
   serve from `cr-landings-v1` (CacheFirst) even while offline.
4. Try a POST request (e.g. publishing a ride) — should fail; this is correct,
   we never cache mutations.

### Forcing cache eviction

Application → Storage → **Clear site data** wipes everything (SW + caches +
storage). Useful when debugging cache-bust regressions.
